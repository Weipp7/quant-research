## Problem Statement

搭建一个供个人使用的 A 股量化研究平台。用户处于量化新手阶段,既没有明确的 edge 假设,也不熟悉现有云量化平台,但同时具有"赚钱"和"学习"两个强动机。

目标是 12 个月内找到 1-2 个在样本外仍有效的策略,或按止损条件停手把资金转入指数基金。

## What Makes This Cool

真正让这件事有趣的不是"系统搭得多漂亮",而是:

1. 每周都能有一个新的回测结果 — 真实的"假设-验证"循环,每一次都在学新东西
2. 你的钱、你的代码、你的策略,全链路掌控
3. 一年后找到了有效策略 → 系统自然从"研究平台"升级为"个人交易系统";没找到 → 学到的量化工程和金融认知仍然是真本事

## Constraints

- **资金**:个人资金,需设置止损上限(时间 + 金额)
- **人力**:单人兼职
- **技能起点**:Python 基础假设存在,量化领域知识较浅
- **监管**:A 股个人账户,程序化交易须通过 QMT/Ptrade 等三方客户端,前期建议只做 paper trading(模拟交易 — 用真实行情但不真下单)
- **数据**:T+1 日频数据免费(AKshare / Tushare 基础版),分钟/Tick 级需付费

## Engineering Disciplines(新增 — /plan-eng-review 产出)

这五条是量化研究平台的**架构级不可妥协**项 — 不是可选优化,是防止新手死法 #1-#5 的底层结构。

### 1. 数据访问纪律 — 防 Lookahead Bias

所有数据加载必须经过显式 as-of 时间点 API:

```python
# utils/data_loader.py
def get_bars(symbol: str, as_of_date: date, lookback_days: int) -> DataFrame:
    """
    在 as_of_date(含)及之前 lookback_days 天的行情。
    架构层保证:T 日的策略决策中,只能看到 T-1 日及之前的数据。
    """
    # SQL WHERE trade_date < as_of_date — 注意严格小于
```

禁止在策略代码中直接 `df.loc[date]`。所有 date 参数必须通过 `as_of_date` 签名。

### 2. 动态成分股 — 防 Survivorship Bias

回测指数时,每个交易日必须用**那一天**的成分股快照,不是当前的:

```python
# utils/constituents.py
def load_index_components(index_code: str, as_of_date: date) -> list[str]:
    """
    使用 Tushare Pro index_weight 或 AKshare index_stock_cons_sina
    返回 as_of_date 那天指数的实际成分股列表。
    空结果必须显式 raise (见 TODO 2)。
    """
```

**Phase 2 本地重现时必须启用**。Phase 1 聚宽上第一个回测可以用当前成分股加速,但代码注释明确标记 "placeholder, Phase 2 起切换动态"。

### 3. 数据版本快照 — 防静默数据变化

DuckDB 里维护一张 `data_ingest_log` 表:

```sql
CREATE TABLE data_ingest_log (
    source TEXT,           -- 'akshare' | 'tushare_pro'
    dataset TEXT,          -- 'daily_bars' | 'index_components'
    symbol TEXT,           -- optional,单股级记录
    download_date DATE,    -- 这次拉取的日期
    rows_count INTEGER,
    checksum TEXT          -- optional,md5 of content
);
```

每次回测输出结果时,同时记录它所引用的数据版本(最近一次 `download_date`)。未来"上周的回测重跑不出来"就能立刻定位到数据变化。

### 4. Notebook 与模块的职责边界

- `notebooks/` **只做探索和可视化**,不定义关键逻辑
- 一旦一段逻辑要被第二个 notebook 复用,**立即**搬到 `strategies/` 或 `utils/` 下的 `.py` 文件,并为它写一个 `pytest` 测试
- Notebook 首行必须是 `%load_ext autoreload` + `%autoreload 2`,保证模块改动立即反映
- 每个 notebook 顶部写明: 目的、输入、产出、对应的 .py 模块

### 5. 依赖锁定

`requirements.txt` 固定版本号,不用 `>=`:

```
akshare==1.x.y
backtrader==1.9.78.123
duckdb==1.x.y
pandas==2.x.y
tenacity==8.x.y  # 用于 D6 重试
```

不引入 poetry/uv 等工具 — pinned pip 对单人项目足够。

## Phase 2.5:回测引擎验证(Engine Validation)— 新增

**产出**:2 个手算金标准(Golden Backtest)测试,常驻 `tests/test_engine.py`。

### Test 1: 买入持有

```python
def test_buy_and_hold_kweichow_moutai():
    """

    什么都不做,持有到 2023-12-31。
    期待:最终 NAV = 手算值 ± 0.01%
    """
    # 1. 手算:查 2020-01-02 收盘价 P0 = X,2023-12-31 收盘价 P1 = Y
    # 2. 除权:期间所有分红 + 拆股按历史事件表调整
    # 3. 手算最终 NAV 写入常量 EXPECTED_NAV
    # 4. 用 Backtrader 跑同一策略,断言 abs(actual - EXPECTED_NAV) / EXPECTED_NAV < 0.0001
```

### Test 2: 每月第一交易日等权买入 N 只

```python
def test_monthly_rebalance_equal_weight():
    """
    每月第一个交易日,把可用现金等权分配到沪深 300 前 5 只。
    2020-01 至 2020-12 一年期。
    期待:最终 NAV = 手算值 ± 0.01%
    """
```

**通过标准**:手算值在 Excel 里算清楚,代码输出与它偏差 < 0.01%。不达标 → 回测引擎有 bug,必须修复才能进 Phase 3。

**这是 "回测对得上聚宽" 声明的前置验证**。本地引擎自己不能给确定答案,谈不上对齐聚宽。

**触发点**:Phase 2 第一次拉沪深 300 全量日线数据时。

**实现**:`utils/data_loader.py` 的 batch 下载函数:

### TODO 2: 成分股空集显式报错

**触发点**:Phase 2 实现 D2 动态成分股时**同步完成**。

**实现**:`utils/constituents.py` 的 `load_index_components(index_code, as_of_date)`:
- 返回空 list 时,先往前回溯到最近一个有效的交易日快照

- 绝不静默返回空 list

**工程量**: 5 行代码。

## Premises

1. 用户的真实目的是"通过同时做量化研究 + 系统工程,最大化个人价值"。赚钱和学习是同等重要的动机
2. 赚钱是学习的下游 — 没有识别和验证 edge 的能力,长期赚钱概率 < 5%。路径必须先学后赚
3. 理性的第一步是"上聚宽/米筐跑出第一个回测"(1-2 周内),而非"搭本地平台"(1-3 月)
4. 本地自建的价值体现在:(a) 云平台做不到的场景,(b) 复杂实验设置,(c) 系统工程学习本身是目标 — 至少需命中一条
5. **项目必须设"止损"** — 时间上限(12 个月)+ 金额上限(用户自填具体数字,建议不超过年收入 5%-10%)+ 产出上限(6 个月内没有 10 个回测结果则重新评估)

## Approaches Considered

### Approach A: 云平台单线程
聚宽/米筐上纯写策略,不触本地代码,月内出 3-5 个回测后再决定。
- 工程量:近乎为零
- Pros:即时反馈;学到的是策略设计这项真正赚钱的技能
- Cons:完全不写系统代码,"搭系统"动机完全未满足;被云平台锁定

### Approach B: 混合渐进(**已选定**)
Week 1-2 聚宽跑第一个策略 + 本地搭最小数据仓。Week 3-4 本地重现云端回测。之后按真需求扩展。
- 工程量:M(1-2 周内有可用本地环境)
- Pros:赚钱+学习双动机同时照顾;本地代码 < 500 行;"回测对得上云端" = 最强正确性验证
- Cons:两边代码要同步;最终可能发现本地部分用不上

### Approach C: 本地研究平台直接自建
Qlib(微软量化框架)或 Vnpy 为骨架,1-3 个月完整搭建。
- 工程量:L(1-3 个月)
- Pros:系统工程覆盖度最全;5 年可用
- Cons:前 1-3 个月无策略输出;新手最容易掉的坑("系统越搭越完美但一个真策略都没跑过")

## Recommended Approach: B) 混合渐进

### Why B
用户明确表态"赚钱 + 学习"双动机。A 满足赚钱不满足学习;C 满足学习不满足赚钱;B 是唯一在两头都有产出的路径。此外,"本地回测对得上云端" 本身就是对"你是否真的看懂回测"的最好考试。

**Phase 1:云端第一个策略(Week 1-2)**

**Phase 3:本地重现云端回测(Week 3-4)**
- 用本地数据 + `backtrader`(或简单自写回测循环)复现 Phase 1 的双均线策略
- 目标:本地回测结果与聚宽偏差 < 1%
- 对不上时,逐项排查:数据(复权方式)、交易成本、滑点模型、停牌/涨跌停处理
- **这一步本身是最有价值的学习** — 让你理解"回测里藏了多少隐藏假设"

**Phase 4:决策点(Week 5+)**

根据 Phase 1-3 的实际体验回答:
- 本地环境是否真的解决了云端解决不了的问题?
- 下一个策略我想在云上跑还是本地跑?
- 是否需要扩展本地环境(加 Tushare Pro、分钟数据、因子库等)?

**原则:只在有具体不满 / 具体需求时扩展本地**。避免为了"完整性"而扩展。

## Success Criteria

**短期(1 个月):**
- ✅ 聚宽跑出至少 2 个不同风格的回测(趋势 + 均值回归)
- ✅ 本地 DuckDB 存有沪深 300 全部成分股的日线
- ✅ 本地回测结果与聚宽对上(误差 < 1%)

**中期(3-6 个月):**
- ✅ 跑过至少 10 个不同假设
- ✅ 有 1-2 个在样本外(out-of-sample 数据 — 回测时没用到的近期数据)仍保持正收益的策略
- ✅ 结构化的"假设-回测-结论"笔记至少 10 条

**长期(12 个月):**
- ✅ 要么找到能持续赚钱的策略,开始真钱小仓位实盘
- ✅ 要么按止损条件停手,把钱换指数基金
- **两个结局都算成功**。失败的定义是:找不到 edge 但还在继续投入时间和金钱
