## 2026-05-08 — order_value 取整导致实际下单比预期少

**现象**：聚宽 `order_value(stock, amount)` 会向下取整到100股整数倍，长期累积后实际持仓和理论值出现偏差。V1 基线版的 88% 收益里有一部分来自这个 bug 带来的"不规范成交"。

**原因**：`order_value` 内部把金额转成股数时向下取整，每次少买几股，但持仓价值计算没有相应修正。

**解决**：手动 `int(amount / price / 100) * 100` 算出整手数后改用 `order(stock, shares)` 下单。

**教训**：聚宽回测里 `order_value` 看似方便，但涉及高精度统计时要换成手动计算手数。修复后（v1_fixed）收益从 88% 降到 86%，才是真实数据。

## 2026-05-08 — 高价股用固定比例建仓会建不了仓

**现象**：10万本金 × 10%仓位 = 1万元，茅台（600519）、平安（601318）等高价股买不到 1 手（100 × 1000+ 元 > 1万），order 不执行。

**原因**：固定的 `position_per_stock` 比例在高价股上分配的资金不够一手。

**解决**：v1_fixed 暴露此问题后，V-Final 改用平均价格50元以下的标的池（平安银行、招商银行、长江电力、神华、恒瑞医药 + 5 ETF），规避高价股问题。

**教训**：设计多股票策略时，先核对所有标的在目标本金下是否够建仓，否则部分标的会从头到尾都是"空仓"。

## 2026-05-08 — ETF 的低波动是网格策略的天敌

**现象**：把10只个股换成10只ETF（V1.1），年化从 6.71% 直接腰斩到 2.36%，夏普变负。加了ATR动态间距（V1.2）也只涨到 2.85%，交易笔数多 21% 但收益几乎不变。

**原因**：网格策略依赖标的在同一价格区间反复震荡来赚取差价。ETF 分散持仓本就是为了降低波动，和网格的盈利逻辑相悖。

**解决**：放弃全 ETF 路线；V-Final 回归 5股+5ETF 混合池，用波动率较高的个股提供主要收益，ETF 作为分散工具。

**教训**：标的选择比参数优化更重要。网格适合有趋势+震荡特征的个股，不适合追求"低波动"的 ETF 组合。

## 2026-05-08 — 仓位管理与网格哲学相互打架

**现象**：V2.0 加入四档仓位管理（STRONG/BALANCED/WEAK/BEAR），弱市降仓但不停止交易。结果年化反而从 6.56% 降到 4.45%，夏普从 0.195 降到 0.043；现金占比频繁 50%+，目标仓位达不到，状态切换过于敏感。

**原因**：网格是"被动应对价格波动"的逻辑，仓位管理是"主动判断市场状态"的逻辑，两者前提假设不同，强行结合导致两套逻辑互相掣肘：弱市时加仓机会被仓位限制掐掉，转强时又踏空。

**解决**：V-Final 完全去掉仓位管理，回归简洁的大盘MA20开关（牛通/熊停）。

**教训**：每加一个"看起来有用"的机制都必须单独测试净效果，不能凭直觉堆功能。简单的机制反而更稳健（"Occam's razor in quant"）。

## 2026-05-08 — 大盘过滤是硬开关，不是仓位管理

**现象**：大盘 < MA20 时策略完全停止交易，资金100%闲置。强弱转换时反应慢，错过反弹初期；长期熊市期间资金躺着贬值。

**原因**：MA20 过滤用的是"昨日收盘 ≤ MA"就全停，没有过渡机制。

**影响**：不是 bug，是设计选择。测试结果表明，对于多标的组合，这个硬开关总体有效，但会错过部分反弹机会。

**教训**：大盘过滤是控制风险的最有效手段，但副作用是"踏空"。接受这个 tradeoff；如果不能接受踏空，就要走仓位管理路线，但仓位管理有自己的问题（见上一条）。

## 2026-05-07 — 本地复现聚宽回测时，收益差距高达 21pp

**现象**：用 qfq 数据 + 收盘价执行复现双均线，本地 -22.75% vs 聚宽 -44.27%，差 21.5pp。

**排查过程**（逐项拆解）：

| 组合 | 收益 | 盈/亏次数 | 胜率 |
|------|------|----------|------|
| 聚宽基准 | -44.27% | 31/78 | 0.284 |
| qfq + 收盘价 | -22.75% | 35/74 | 0.321 |
| qfq + 开盘价 | -28.17% | — | — |
| raw + 收盘价 | -29.91% | 30/77 | 0.280 |
| **raw + 开盘价** | **-28.91%** | **31/76** | **0.290** |

**原因拆解**：
1. **复权方式（最大影响）**：qfq 用 2026 年作为基准，把 2020 年价格往下调了 6 年分红。信号触发时机与聚宽动态复权（每日以当日为基准）不同，导致胜率和盈亏对数都有偏差。换用 raw 不复权数据后，胜率（0.280~0.290）和盈亏对数（31 盈）与聚宽吻合。
2. **执行价格**：收盘 → 开盘，额外缩小约 5pp 差距。
3. **剩余约 15pp 无法解释**：最可能来源是聚宽与新浪数据源的价格差异（尤其是开盘价），以及聚宽真实执行时点（09:30 后几分钟）不等于精确开盘价。

**教训**：
- 复权方式是回测中最隐蔽的假设。研究信号时机要用 raw（还原历史真实价格），研究长期收益时要用 hfq（累积复权正确）。qfq 适合画图，不适合复现历史回测。
- 平台级复现（聚宽 → 本地）在没有完全相同数据源时，信号时机可以复现（±1~2 次交易），但绝对收益无法精确对齐，15pp 级别的误差是正常的。
- 不要因为本地结果"更好"就以为策略改进了。可能只是数据假设不同。

## 2026-05-07 — akshare 东财源在 WSL 下经常连接被拒

**现象**：`ak.stock_zh_a_hist()` 报 `ConnectionError: Remote end closed connection without response`，但 `curl --noproxy '*'` 直接访问该 URL 偶尔能通、偶尔也不通。

**原因**：东财 API 对 Python `requests` 的 TLS 指纹敏感，且本身有限流/WAF。WSL 继承了宿主的 Clash 代理（`HTTP_PROXY / HTTPS_PROXY / ALL_PROXY`），代理对境内数据源反而造成干扰。

**解决**：
1. 切换到新浪源 `ak.stock_zh_a_daily()`，稳定性好很多。
2. 在 `utils/data_loader.py` 启动时把 proxy env 清空（保留 `QR_KEEP_PROXY=1` 开关）。

**教训**：
- akshare 有多个后端，不同函数调不同数据源。遇到连接问题先换函数，而不是折腾网络配置。
- WSL 的代理对境内金融 API 帮倒忙，在数据层入口统一清掉最省心。

## 2026-05-07 — 新浪源的股票代码格式与聚宽不同

**现象**：直接把聚宽的 `"000001.XSHE"` 传给 `ak.stock_zh_a_daily()` 报错。

**原因**：新浪源要求 `"sz000001"`（深交所加 sz 前缀）或 `"sh600519"`（上交所加 sh 前缀）。

**解决**：`data_loader._to_sina_code()` 根据首字符自动判断交易所并加前缀。

**教训**：不同平台的股票代码格式是个长期摩擦点，统一在数据层转换，策略层只用 6 位数字代码。

## 2026-05-07 — GBK 编码导致 CSV 在 GitHub 上乱码

**现象**：聚宽导出的 `result_1.csv` 在 GitHub 网页端显示乱码。

**原因**：聚宽 CSV 是 GBK 编码，GitHub 默认 UTF-8 渲染。

**解决**：入库时用 `iconv -f gbk -t utf-8` 转换，只存 UTF-8 版本。

**教训**：境内金融平台导出文件几乎都是 GBK，入库第一步先转码。

## 2026-05-07 — mplfinance 中文标题乱码

**现象**：K 线图标题里的中文变成方块（"□□ 60 □□□□"）。

**原因**：matplotlib 默认字体 DejaVu Sans 不含 CJK 字形。

**解决**：改用英文标题（简单） 或 安装 Noto CJK 字体后设置 `plt.rcParams["font.family"]`（完整解法）。

**教训**：Linux 上用 matplotlib 做中文图表，要么装 `fonts-noto-cjk`（`sudo apt install fonts-noto-cjk`），要么坚持英文标题。
